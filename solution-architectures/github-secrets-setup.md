# KinCare — GitHub Secrets & Variables Setup

Configure these in **Settings → Secrets and variables → Actions** on the GitHub repo.

---

## Repository Secrets (encrypted, all environments)

| Secret | Description |
|---|---|
| `VERCEL_TOKEN` | Vercel personal access token (vercel.com → Settings → Tokens) |
| `VERCEL_ORG_ID` | Found in vercel.com → Settings → General → Team ID |
| `VERCEL_PROJECT_ID` | Found in vercel.com → Project → Settings → General → Project ID |
| `SLACK_WEBHOOK_URL` | Incoming webhook URL for deploy failure alerts (optional) |

Create at **Settings → Environments → New environment → beta**

### Secrets

| Secret | Description |
|---|---|
| `BETA_DB_CONNECTION_STRING` | Neon beta branch connection string |
| `RAILWAY_BETA_TOKEN` | Railway project token for beta service |
| `E2E_TEST_EMAIL` | Email of a seeded beta test account |

|---|---|

## Environment: `production`

Create at **Settings → Environments → New environment → production**

Enable **Required reviewers** — add at least one person who must approve before
the production deploy job runs. This prevents accidental deploys.

### Secrets

|---|---|

## Neon database branching (recommended)

Neon supports instant DB branches — use them to isolate beta from production:

1. Create project in Neon: `kincare`
2. `main` branch → production DB
3. Create branch `beta` from `main` → beta DB
4. Use branch connection strings in the corresponding GitHub environment secrets

This gives beta its own isolated DB that can be reset without affecting production.

## Vercel project setup (one-time)

```bash
# From repo root
cd src/KinCare.Web
npx vercel link        # links to existing or new Vercel project
npx vercel env add KINCARE_API_URL   # add per environment
```

After linking, copy the `VERCEL_ORG_ID` and `VERCEL_PROJECT_ID` from `.vercel/project.json`
into the GitHub repository secrets.
